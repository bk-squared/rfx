# rfx Validation Evidence

Everything in this tree is the repo's cross-validation EVIDENCE: scripts we
run to verify rfx against analytic references and other solvers. They are kept
public for transparency, but they are measurement fixtures with case-specific
evidence boundaries — new users should start with the learning path in
`examples/README.md`, not here.
`crossval/manifest.json` is the machine-readable registry. Images generated by
these scripts are cached next to them in `crossval/` and under
`../examples/nonuniform_patch_demo/` for documentation builds.

## Ordering principle

Scripts are grouped by the **kind of evidence** they produce, then by
the technical complexity inside each group:

- **A — primitives**: short validations against a known analytic or
  single-solver reference.  Each one pins a single FDTD primitive
  (propagation, resonator, flux, multilayer R/T).
- **B — engineering validation and diagnostics**: full RF-engineering
  geometries and regression cases with case-specific evidence boundaries.
- **C — inverse design**: differentiable forward + `jax.grad` loop.
- **D — non-uniform workflows**: practical demos that exercise the NU
  mesh runner (graded dz, xy-graded variants).

Within each group the filename numbering reflects implementation
complexity, not the evidence tier.

## Research-only subgrid examples

- `research/subgrid/12_subgrid_disjoint_prototype.py` is a short-window smoke
  for the Stage-2 disjoint topology, which is not long-time energy-stable.
- `research/subgrid/13_subgrid_material_validation.py` demonstrates the guarded
  fail-closed subgrid boundary and checks the material setup on the uniform
  analytic reference lane.

These scripts are preserved research artifacts, not crossvals or public feature
claims. SBP-SAT/subgridding remains outside public support scope.

## CI / runtime tier

A/B grouping describes case family, not evidence strength or execution
frequency. The manifest records
whether each crossval runs in the CPU runner, the scheduled external-reference
workflow, or a GPU/VESSL/manual lane. C- and D-tier scripts are demos. Research
scripts run manually under `python validation/research/...`; their inline gates
do not promote them into crossval or the public support surface.

## Picking what to cite

- **Public accuracy claims**: cite the per-case evidence, not the group letter.
  External OpenEMS / Meep / Palace comparisons are stronger than self-invariant
  regression locks; analytic oracles remain direct physics evidence inside
  their stated envelope.
- **Practical workflow demos**: start with D1 for non-uniform meshes
  and C3 for gradient-based optimisation against an analytic reference.
- **First script a new reader should run**: A1
  (`03_straight_waveguide_flux.py`) — fastest, least setup.

## VESSL runners

Selected scripts are launched on GPU CI via YAMLs in `scripts/vessl_*.yaml`
(gitignored at rest; force-added per file).  The patch-antenna job
(B1) is the primary GPU physics pin.

## What's *not* here

Earlier revisions of the examples tree had `40_accuracy_validation/` and
`50_advanced/` directories.  Their `.py` sources were removed during
the crossval audit (commit `0211800`); the orphan PNGs were deleted in
the Issue #35 cleanup.  No scripts exist for these families today.

## tap_paper — projected targets, not committed results

`tap_paper/` holds the paper-support scripts. Their full-resolution headline
numbers are projected targets, not validated claims; see `tap_paper/README.md`
for per-script status.
